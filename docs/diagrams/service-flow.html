NaxOne · 服务启动流程

用户点击「启动 Nginx」 → Nginx/Apache 互斥处理 → PHP-CGI 联动启动 → 状态广播

1
用户操作 User
在仪表板点击 Nginx 卡片的 ▶ 启动按钮（或点 svc-toggle 滑块）。
UI 立刻把按钮置 busy，禁用重复点击；pauseAutoRefresh() 暂停 5s 状态轮询，避免显示中间态。
▼
2
前端发起 IPC 调用 Vue 3
Dashboard.vue 调 invoke("start_service", { id }) 进入 Tauri Command 层。
10s 超时保护（invokeWithTimeout）；失败时 friendlyError() 把后端错误翻译成人话 toast。
▼
3
Tauri Command 进入 ServiceManager naxone-tauri
commands/service.rs 拿到 AppState，把 target 和 others（同 kind 其他实例）传给 service_manager.start_with_deps()。
写 push_log 进 NaxOne 自己的活动日志（前端可见）；同时 emit services-changed 让其他页面刷新。
▼
4
互斥处理：先停 Apache naxone-core
ServiceManager 检查 others 中是否有运行中的 Apache（与 Nginx 共享 80 端口）。如有，先 stop 再 start。
同 kind 多版本（如 PHPStudy 自带 Nginx 1.15 + Store 装的 1.25）也要停掉非 target 的版本，避免端口抢占。
▼
5
Spawn nginx.exe + 校验配置 NativeProcessManager
用 tokio::process::Command spawn，creation_flags(CREATE_NO_WINDOW) 不弹黑框；启动后 500ms 试 try_wait()，若已退出则读 stderr 报具体错。
PID 失败兜底：child.id() 拿不到时降级用 netstat 反查端口对应进程。状态同步写到 status_cache，前端拉新值。
▼
6
联动启动 PHP-CGI naxone-core
遍历 others 中所有 PHP 实例，未运行的挨个 spawn（每个独立端口 9001/9002/...），并启 watchdog 监听崩溃自动重启。
watchdog 加了 backoff（2s→4s→8s→16s→32s），超过 5 次崩溃停止重启避免死循环。
▼
7
回调前端 + Toast 通知 Vue 3
Command 返回最新 services 数组（含 Nginx + 联动起来的 PHP-CGI 状态）。前端 splice 更新本地 ref，toast.success("已启动 Nginx + 3 个 PHP-CGI")。
5s 后 svcTimer 自动 refresh 一次状态，确认 PID/内存数据是最新的。
